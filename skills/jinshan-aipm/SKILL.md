# 金山AIPM

构建一条沿用 `designPM-ZJY` 思路、但把中间产物收束进一组关联 HTML 交付的产品设计流水线。保持阶段顺序不变：

```text
头脑风暴 -> 提示词增强 -> PRD -> 线框图 -> 高保真原型
```

变化点不在流程，而在交付物：

- 不默认保留独立的 `00-...md`、`01-...md`、`03-...md`
- 从阶段 2 开始，围绕 PRD HTML、原型 HTML、统一 HTML 三个交付物持续迭代
- 阶段 2 写入 HTML 之前，必须先基于前序结论调用 `prd`，产出一份完整标准 PRD，再压缩映射回 HTML 的 PRD 视图
- 头脑风暴结论与增强提示词作为内部推导输入，最终只沉淀为更完整的 PRD 交付内容
- 线框图与交互规范融入 PRD 视图时，默认直接复用原型中的真实交互片段，只允许为讲解目的做范围裁剪，不允许退化成静态示意图或单独重画一套假交互
- 最终统一页顶部必须提供两个按钮：`查看PRD`、`查看原型`
- 最终必须同时交付 3 个 HTML：独立 PRD、独立原型、PRD+原型统一页

## 核心原则

- 保持阶段顺序，不要跳过头脑风暴、提示词增强或线框图阶段。
- 不把“融入 PRD”理解成“省略过程”；每个阶段都要做，但并非每个阶段都要在最终 PRD 页面显式露出标题。
- 从阶段 2 开始，把完整标准 PRD 视为内容事实源，把 3 个 HTML 视为不同展示层。
- 阶段 2 的 HTML PRD 不是凭空概括，而是必须建立在一份由 `prd` 生成的完整标准 PRD 之上。
- `prd` 生成的完整标准 PRD 是内容事实源，HTML 中的 PRD 视图是展示层；前者更完整，后者更适合评审与演示。
- 阶段 3 不再单独输出线框图文件，而是在 PRD 视图内放入直接来自原型的交互片段与说明。
- 阶段 4 需要同步维护独立原型页与统一页，不允许只产出统一页。
- 所有面向用户的内容默认使用中文。
- PRD 页必须锁定为金山风格的纯色红白体系：白底、红色强调、深色正文，不允许漂移为蓝白、紫白、黑金、渐变科技风或大面积图片底。
- PRD 页允许少量深灰 / 暖灰作为信息层级辅助，但不得改变红白为主的视觉判断。
- PRD 页中凡是承担“配图 / 示意 / 流程解释 / 页面说明”职责的模块，默认都应为可操作的动态交互片段，并与原型页复用同一套状态逻辑。
- 只要技术上可行，PRD 说明区与原型区应优先复用同一套组件、事件处理、状态源与数据结构，不要分别实现两套近似交互。
- 原型页则根据具体产品题材决定配色策略。
- 如果用户明确要求额外导出 Markdown 版本，再额外生成；否则以 HTML 主交付为准。

## 统一交付物

默认固定维护三个交付文件名：

```text
workspace/
├── 02-产品需求文档.html
├── 03-高保真原型.html
└── 04-PRD与原型统一.html
```

执行方式：

1. 阶段 2 先基于前序结论调用 `prd`，形成完整标准 PRD
2. 创建 `02-产品需求文档.html`，把完整 PRD 压缩映射为独立的 HTML PRD 视图
3. 阶段 3 继续完善 `02-产品需求文档.html`，补齐直接复用原型片段的线框图、流程图、交互规范
4. 阶段 4 创建并完善 `03-高保真原型.html`，承载完整可演示原型
5. 基于前两者再生成 `04-PRD与原型统一.html`，在一个页面中用按钮切换 PRD 与原型
6. 交付时必须同时给出这 3 个 HTML，不允许缺任一项

如果需要保留过程文件，可额外保留完整标准 PRD 的 Markdown 或文本稿；但默认最终交付检查对象始终是上述 3 个 HTML。

## 阶段流程

### 阶段 0：需求头脑风暴

使用 `requirement-brainstorming-zjy` 的方法进行结构化澄清，至少覆盖：

- 产品一句话定义
- 目标用户与关键场景
- 核心价值与首版目标
- P0/P1/P2 与明确不做
- 页面清单、导航结构、任务流
- 视觉方向、品牌气质、参考产品

输出要求：

- 可以先形成临时分析内容，但不要把它作为默认最终交付
- 这些内容只作为后续 PRD 与原型设计的内部输入，不要求在最终 PRD 页面显式出现“需求头脑风暴”标题
- 在用户未确认前，不进入后续阶段

### 阶段 1：提示词增强

使用 `enhance-prompt` 的方法，把阶段 0 的结论整理成更专业、更具体的中文设计提示词。

输出要求：

- 产出面向设计生成的高质量中文提示词
- 这些提示词用于约束 PRD 结构、线框表达和原型气质，不要求在最终 PRD 页面显式出现“增强提示词”章节
- 不默认额外保留 `01-增强提示词.md`

### 阶段 2：先完整 PRD，再落 HTML PRD

这一阶段必须分成两个动作，不能跳过其中任意一个：

1. 先把阶段 0 和阶段 1 的结论整理成 `prd` 可直接使用的输入摘要
2. 显式调用 `prd`，先生成一份完整标准 PRD
3. 再创建 `02-产品需求文档.html`，把完整 PRD 压缩为独立可读、可汇报的 HTML PRD 视图

这里的关键不是“顺手参考一下 PRD 写法”，而是：

- `prd` 必须真正参与 PRD 阶段
- HTML 中的 PRD 内容必须尽量继承 `prd` 产出的结构完整性
- 不能只保留几块很薄的概览卡片，导致标准 PRD 的关键信息被丢失

PRD 视图至少包含这些真正用于交付的内容：

1. 项目概览
2. 产品需求文档
3. 信息架构
4. 页面清单
5. 功能范围与优先级
6. 关键交互说明
7. 线框图与页面示意
8. 流程图
9. 风险与待确认项

此外，HTML 中的这些内容应尽可能映射并体现 `prd` 的标准结构，例如：

- `Executive Summary` -> 项目概览 / 问题、方案、成功标准
- `User Experience & Functionality` -> 用户角色、用户故事、验收标准、非目标
- `AI System Requirements`（如适用） -> 工具依赖、评估方式
- `Technical Specifications` -> 架构概览、集成点、安全与隐私
- `Risks & Roadmap` -> 风险、阶段规划、后续版本

如果完整标准 PRD 中存在这些内容，HTML 的 PRD 视图应尽量体现出来，而不是只保留视觉概述。

此时可以暂不生成最终统一页，但必须为后续统一页保留明确的 PRD/原型双视图切换结构。

推荐基础结构：

```html
<header class="aipm-topbar">
  <button type="button" data-view-target="prd">查看PRD</button>
  <button type="button" data-view-target="prototype">查看原型</button>
</header>

<main class="aipm-shell">
  <section id="prd-view">...</section>
  <section id="prototype-view" hidden>...</section>
</main>
```

#### 阶段 2.1：`prd` 调用规则

调用 `prd` 前，先把前序输入整理为简洁但完整的上下文包，至少包含：

- 产品一句话定义
- 目标用户与关键场景
- 核心目标与成功判断
- P0 / P1 / P2 与明确不做
- 页面清单与主任务流
- 技术约束、时间约束、品牌 / 视觉约束（如已知）

调用 `prd` 时，要求它产出一份完整标准 PRD，而不是“适合 HTML 页面展示的压缩版”。

如果用户没有给足信息，而 `prd` 的要求是必须追问关键信息，则应先补齐必要问题，再进入完整标准 PRD 生成。

#### 阶段 2.2：完整 PRD 到 HTML PRD 的映射规则

把 `prd` 产出的完整标准 PRD 压缩进 HTML 时，遵循以下映射原则：

- 保留信息，不保留冗长段落：优先提炼成可读卡片、结构块、摘要列表、流程图与说明组合
- 保留标准 PRD 的骨架：不要只剩视觉稿，必须让读者能看出这是一份真实 PRD 的页面化表达
- 保留可执行信息：成功标准、验收条件、非目标、风险、阶段规划等不应全部丢失
- 保留和原型强相关的部分：用户故事、关键流程、页面职责、状态说明要能直接映射到后续原型片段，并回嵌到 PRD 对应说明区
- 允许合并章节，但不允许关键字段完全消失
- 对于需要动态图示的章节，优先从原型模块反向裁剪出 PRD 讲解版本，而不是先画 PRD 图、再单独补原型。

推荐的 HTML PRD 映射如下：

- `项目概览`
  - 问题陈述
  - 方案摘要
  - 成功标准
- `产品需求文档`
  - 用户角色
  - 用户故事
  - 验收标准
  - 非目标
- `信息架构 / 页面清单`
  - 页面职责
  - 页面关系
- `功能范围与优先级`
  - P0 / P1 / P2
  - 版本边界
- `关键交互说明 / 流程图`
  - 主任务流
  - 状态流
- `技术与实现约束`
  - 架构概览
  - 集成点
  - 安全与隐私
- `风险与待确认项`
  - 技术风险
  - 业务风险
  - 路线图 / 后续阶段

如果完整标准 PRD 很长，HTML 页面应优先做“结构化压缩”，而不是删到只剩两三行抽象描述。

### 阶段 3：把原型片段融入 PRD

继续编辑 `02-产品需求文档.html`，不要创建独立的 `03-线框图与交互规范.md`。
阶段 3 仍然只增强 PRD 页，不要误把这一阶段做成原型页。

把原型解释片段写入 PRD 视图的专属章节，例如：

- `线框图与交互规范`
- `核心任务流`
- `页面跳转关系`
- `组件状态说明`

这些章节应优先复用完整标准 PRD 中已经定义好的用户故事、验收标准、状态约束和页面职责，并直接复用原型中的对应交互模块，而不是重新发明一套脱离 PRD 的静态说明图。

PRD 中的图示不要只做低保真占位块，也不要只做“像原型”的静态页面示意；默认应直接截取、裁剪或嵌入原型中的真实交互模块，让人能在 PRD 里就操作对应状态与流程。

工程实现上，优先采用以下复用方式：

- 同一组件，不同容器：原型页完整挂载，PRD 页挂载裁剪版或限定态版
- 同一状态源：PRD 说明区与原型区共用同一份本地状态、Mock 数据、状态机或事件分发逻辑
- 同一交互脚本：tab 切换、筛选、弹窗、步骤流转、表单反馈不要各写一套
- 同一数据结构：字段名、状态枚举、步骤编号、页面标识保持一致，避免 PRD / 原型两边出现不同语义
- 如必须为 PRD 做展示特化，优先做样式层包装或参数裁剪，不要复制后改造成另一套组件

推荐表达方式：

- 直接嵌入原型中的局部交互模块，作为 PRD 讲解图
- 复用原型中的 tab、筛选、弹窗、状态切换、流程步骤器，作为说明区里的动态示意
- 在不破坏讲解节奏的前提下，对原型片段做局部裁剪、缩放或简化布局
- 结构化列表加状态表，作为动态演示模块的文字补充

无论采用哪种方式，都必须描述清楚：

- 页面结构和区块层级
- 点击 A -> 显示 B 的交互路径
- 组件状态、错误态、空态、加载态
- 页面之间的导航与转场逻辑

注意：

- PRD 中的图示模块应优先直接复用原型代码、状态机、交互路径或组件片段，再按讲解范围做裁剪
- 目标不是做另一套独立设计，而是让 PRD 页面里的图示本身就是原型的一部分
- 如果 PRD 图示与原型区在相同行为上出现不一致，以原型交互逻辑为准，再回头修正 PRD 说明区

### 阶段 3.1：在 PRD 中加入流程图

如果业务流程、审批流、用户路径或页面切换关系较复杂，优先在 PRD 视图中加入可交互流程模块，而不是静态流程图。

推荐顺序：

1. 优先复用原型中的流程步骤器、状态流转组件或可点击路径模块
2. 其次使用纯 HTML + CSS + JS 的可交互流程节点
3. 如使用 SVG / Mermaid，也必须补上点击、切换、展开或状态反馈，不接受纯静态展示

可交互流程模块适合表现：

- 用户主任务路径
- 信息流 / 状态流
- 业务审批流
- 页面跳转流程

流程图视觉建议：

- 默认跟随 PRD 页的红白主视觉体系
- 节点、按钮、状态反馈、说明块风格统一，不要做成工程工具默认蓝灰图
- 保证它既是交付稿中的方案图，也是可操作的讲解组件，而不是临时分析图

### 阶段 4：同一 HTML 中实现高保真原型

创建并完善独立的 `03-高保真原型.html`，同时再生成一个 `04-PRD与原型统一.html` 用于双视图切换展示。

阶段 4 的执行策略改为：

```text
步骤1：以 02-产品需求文档.html 为基础，确认 PRD 页内容完整
步骤2：创建 03-高保真原型.html，并逐页填充高保真界面与交互
步骤3：接入真实图片、状态切换、导航和局部动效，保证原型可独立演示
步骤4：把原型中的关键交互片段回嵌到 02-产品需求文档.html 的对应说明区
步骤5：检查 PRD 说明区与原型区是否复用同一套组件、脚本、状态逻辑，尽量消灭重复实现
步骤6：再创建 04-PRD与原型统一.html，把 PRD 页和原型页整合到同一壳层下
步骤7：检查统一页中的 PRD / 原型切换、响应式、可演示性，以及 PRD 说明图是否可交互
步骤8：交付时同时输出 02、03、04 三个 HTML
```

原型视图必须满足：

- 每个按钮、链接、标签页、弹窗都有行为
- 表单有输入、校验、反馈
- 状态切换可见，且符合 PRD 中定义的交互逻辑
- 无死链、无空点击、无假入口
- 可以直接用于演示

## HTML 一体化规范

统一 HTML 至少包含以下壳层：

- 顶部切换条：`查看PRD`、`查看原型`
- PRD 视图：承载真正交付的 PRD 内容、线框图、流程图
- 原型视图：承载高保真交互页面
- 共用样式系统：颜色、字体、间距、卡片、按钮、标签、弹窗样式
- 共用脚本：视图切换、路由、Mock 数据、局部状态管理
- PRD 说明区与原型区应尽量复用同一套组件、脚本和状态逻辑，避免维护两套行为不一致的图示模块
- 如项目结构允许，优先抽出共享组件层 / 共享脚本层 / 共享状态层，再分别挂载到 PRD 与原型两个视图

建议行为：

- 默认进入 `PRD` 视图，便于先讲方案，再切到原型
- 点击按钮时切换高亮状态，并同步 `aria-pressed` 或 `aria-selected`
- 可使用 hash、query 或本地状态保存当前视图
- PRD 视图允许较长滚动阅读，原型视图更强调沉浸式展示
- 统一页中的 PRD 视图必须与 `02-产品需求文档.html` 的红白规范保持一致，不允许在统一页里换皮

## PRD 页视觉规范

PRD 页不是分析草稿，而是正式交付页面，默认遵循以下视觉要求：

- 主色必须为红白纯色体系，适配金山品牌气质
- 背景必须以白色或浅红色纯色块为主，不允许使用大面积渐变、照片底、玻璃拟态或发光科技风背景
- 强调色必须优先使用金山语境下的红色系，不允许把主强调色替换成蓝、紫、荧光色
- 允许少量深灰或暖灰辅助信息层级，但不要回到默认蓝白科技风
- 标题区、分组卡片、流程图节点、线框示意都要统一在红白体系下
- PRD 页应更像“对内评审 / 对上汇报的视觉方案页”，而不是纯文档排版
- 如果用户没有特别指定，PRD 页优先使用白底、红色强调、浅红分组背景、深色正文的组合
- 如果生成结果中的 PRD 页出现其他主题色主导，必须视为不合格并重做
- 即便图示模块直接复用原型片段，嵌入 PRD 时也必须做红白化外观适配，保证交互逻辑一致、视觉归属仍然属于 PRD 页

## 图片与视觉素材

如果产品题材适合真实图片，优先为原型视图的关键视觉位接入真实图片。推荐位置：

- 首页 Hero
- 榜单或专题封面
- 卡片封面图
- 详情页头图 / 轮播图
- 收藏页缩略图

优先使用同目录脚本：

- `scripts/pexels-search.js`

示例：

- `node scripts/pexels-search.js "moody coffee shop interior"`
- `node scripts/pexels-search.js "editorial restaurant interior" --per-page 6 --orientation portrait`
- `node scripts/pexels-search.js "night city bar warm light" --write pexels-results.json`

选图原则：

- 用英文、场景化、摄影语义明确的搜索词
- 同一页面图片风格统一
- 避免廉价海报感、强广告感和过度摆拍
- 如果网络受限或 Pexels 不可用，再退回渐变、SVG 或 Mock 图

## 质量门槛

在宣布完成前，确认以下事项：

- 阶段顺序保持不变
- 头脑风暴和增强提示词已经被消化进最终交付，但不在 PRD 页上生硬露出过程模块
- 顶部两个按钮可在 PRD 和原型之间切换
- PRD 视图具备可视化表达，而不是纯长文
- PRD 视图中的内容能明显看出来自一份完整标准 PRD，而不是只有轻量概览
- PRD 中承担说明职责的图示模块，必须直接复用原型交互片段，不接受静态图或单独画的伪原型图
- PRD 页满足金山适配的红白视觉方向
- 必须同时产出 `02-产品需求文档.html`、`03-高保真原型.html`、`04-PRD与原型统一.html`
- `04-PRD与原型统一.html` 中的 PRD 和原型需分别可切换、可独立阅读，不是简单拼接长页
- PRD 说明区与原型区在相同交互上的按钮有效性、状态枚举、筛选逻辑、流程顺序、弹窗内容不得出现行为不一致
- 全部界面文案默认是中文
- 最终文件可直接演示，不需要额外解释“这里还没做”

## 示例产物结构

推荐把 PRD 视图组织为：

```text
PRD 视图
├── 项目概览
├── 产品需求文档
├── 信息架构
├── 页面清单 / 功能优先级
├── 线框图与页面示意
├── 核心流程图
└── 风险与待确认项
```

推荐把原型视图组织为：

```text
原型视图
├── 全局导航 / 顶栏
├── 核心页面 A
├── 核心页面 B
├── 表单 / 弹窗 / 抽屉
├── 空态 / 加载态 / 错误态
└── 演示级交互逻辑
```

## 何时不使用这个技能

以下情况不要优先使用 `jinshan-aipm`：

- 用户只要纯 PRD Markdown，不要 HTML
- 用户只要单个页面原型，不关心 PRD 共存
- 用户只要线框图，不需要原型
- 用户只要文案建议，不需要完整设计流水线

这些场景下，应改用更窄的技能，例如 `prd`、`wireframe-prototyping` 或 `frontend-design`。
