# 金山 AIPM HTML 设计流水线

本 skill 用于把用户的初步产品想法转化为可评审、可演示、可交付的中文产品设计产物。\
它继承 `designPM-ZJY` 的核心方法：**需求头脑风暴 → 提示词增强 → PRD → 原型图 / 线框图 → 高保真 Demo**，但最终交付物统一收束为 **3 个 HTML 文件**。

## 一、核心目标

将用户的模糊需求转化为：

1. 一份可视化 PRD HTML，用于产品方案评审；
2. 一份高保真可交互 Demo HTML，用于演示产品体验；
3. 一份 PRD + Demo 统一 HTML，用于在同一个页面中切换查看 PRD 与 Demo。

最终文件必须能直接在浏览器中打开，不依赖后端服务，不需要额外解释。

## 二、固定交付物

默认只交付以下 3 个 HTML 文件：

```text
workspace/
├── 02-产品需求文档.html        # PRD 可视化页面，红白配色，金山风格
├── 03-高保真Demo.html          # 高保真可交互 Demo 原型
└── 04-PRD与Demo统一.html       # 同一页面中切换查看 PRD 与 Demo
```

说明：

- `00-需求头脑风暴设计.md`、`01-增强提示词.md`、`03-线框图与交互规范.md` 不作为默认最终交付物。
- 头脑风暴、提示词增强、线框图与交互规范仍然必须执行，但默认作为内部推导过程，被吸收进最终 3 个 HTML。
- 如果用户明确要求保留过程文档，再额外输出 Markdown 或文本版本。
- 交付时必须同时给出 3 个 HTML，不允许只给其中一个或两个。

## 三、工作流程总览

```text
用户需求
  ↓
阶段 0：需求头脑风暴
  ↓
阶段 1：提示词增强
  ↓
阶段 2：完整 PRD 生成
  ↓
阶段 3：原型图 / 线框图 / 交互规范设计
  ↓
阶段 4：生成 02-产品需求文档.html
  ↓
阶段 5：生成 03-高保真Demo.html
  ↓
阶段 6：生成 04-PRD与Demo统一.html
```

核心原则：

- 阶段顺序不能跳过。
- 阶段 0 是硬门禁：没有完成需求澄清和用户确认前，禁止创建或修改任何最终 HTML 文件。
- 中间推导可以不显式落成独立文件，但必须真实参与后续输出。
- 最终 PRD HTML 不是简单摘要，而是完整 PRD 的可视化表达。
- 最终 Demo HTML 不是静态截图，而是可点击、可输入、可切换状态的高保真交互原型。
- 最终统一 HTML 不是两个页面的简单拼接，而是具备清晰切换壳层的综合交付页。

## 四、阶段 0：需求头脑风暴

沿用 `requirement-brainstorming-zjy` 这个skill进行需求头脑风暴方式，通过结构化对话识别用户真实意图。

### 必须澄清的内容

至少覆盖以下问题：

1. 产品一句话定义：这是一个什么产品？
2. 目标用户：谁会使用？
3. 使用场景：用户在什么情况下打开它？
4. 核心问题：它解决什么痛点？
5. 首版目标：MVP 版本要先解决什么？
6. 功能范围：P0 / P1 / P2 功能分别是什么？
7. 明确不做：当前版本不做什么？
8. 页面结构：需要哪些核心页面？
9. 任务流：用户从进入到完成核心任务的路径是什么？
10. 交互要求：哪些按钮、弹窗、筛选、表单、状态必须可用？
11. 视觉方向：是否有品牌、配色、参考产品或风格约束？
12. 成功指标：如何判断这个产品方案有效？

### 执行要求

- 如果用户信息不足，只能先问必要问题，禁止直接生成、禁止先写 HTML、禁止用大段自拟假设替代澄清。
- 问题应尽量少而关键，优先补齐会影响 PRD 和 Demo 的信息；每轮建议问 1-2 个问题。
- 至少问够10-15轮以上，彻底了解清楚用户的需求和对产品和demo的所有想法
- 可以给用户 2-3 个产品方向方案供选择。
- 用户确认方向后，才能进入下一阶段。
- 头脑风暴结论作为后续 PRD 和 Demo 的事实输入，不默认在最终页面中以“头脑风暴记录”形式生硬展示。

### 阶段 0 通过标准

进入阶段 1 前，必须已经得到用户明确回答或确认以下最低信息：

1. 产品定位：产品是什么，不是什么；
2. 目标用户：核心用户群、供给方或内容生产者、管理/运营角色；
3. 核心场景：用户为什么打开、要完成什么任务；
4. MVP 范围：P0 功能、明确不做、首版成功标准；
5. 页面与任务流：首页/主工作区/详情/发布或创建/个人中心等核心页面，以及主路径；
6. 关键交互：至少列出需要可演示的按钮、输入、筛选、弹窗、状态切换；
7. 视觉方向：红白 PRD 固定，Demo 的产品风格需由用户确认或给出可确认选项。

如果用户只说“做一个 XX Web / App / 产品”，这不满足通过标准。必须先发起澄清问题，并等待用户回复。

### 澄清问题格式

当需要澄清时，只输出问题，不创建文件，不给最终交付清单。推荐格式：

```text
在开始生成 HTML 前，我需要先把需求定准。请先确认这几项：

1. ...
2. ...
3. ...
```

问题要围绕会改变 PRD 和 Demo 的决策，不要询问无关偏好。

## 五、阶段 1：提示词增强

沿用 `enhance-prompt `skill的提示词增强方式，把阶段 0 的结论转成适合生成 PRD、原型图和高保真 Demo 的设计提示词。

### 增强重点

增强后的提示词应明确：

1. 产品定位；
2. 用户角色；
3. 核心任务流；
4. 页面清单；
5. 功能优先级；
6. 关键交互；
7. 信息架构；
8. 数据对象；
9. 页面状态；
10. 视觉风格；
11. PRD 表达方式；
12. Demo 交互要求。

### 执行要求

- 提示词增强结果不默认单独输出为 `01-增强提示词.md`。
- 增强结果必须被用于约束 PRD 结构、原型图逻辑和 Demo 视觉。
- 不允许只写泛泛的 UI 提示词，必须能直接指导页面与交互生成。

## 六、阶段 2：完整 PRD 生成

在写入 HTML 之前，必须先形成一份完整的标准 PRD。沿用 `prd` 这个skill去写需求文档\
HTML PRD 是展示层，标准 PRD 是内容事实源。

### 标准 PRD 至少包含

1. 项目背景与问题定义；
2. 产品定位与目标；
3. 目标用户与核心场景；
4. 用户故事；
5. 功能范围与优先级；
6. 页面清单；
7. 信息架构；
8. 核心业务流程；
9. 关键交互规则；
10. 数据对象与字段；
11. 状态、异常态、空态、加载态；
12. 指标与验收标准；
13. 风险与待确认项。

### 执行要求

- 必须先生成完整 PRD，再将其映射成 HTML PRD。
- 不允许直接写一个只有概览卡片的轻量 PRD 页面。
- PRD 内容要能支撑 Demo 的每一个页面和交互。
- 如果是 AI 产品，还必须补充模型能力边界、输入输出、失败兜底、评估方式和人工干预机制。

## 七、阶段 3：原型图 / 线框图 / 交互规范设计

本阶段不默认输出独立线框图文档，而是把线框图、页面结构和交互规范融入 PRD HTML 与 Demo HTML 中。

### 必须设计的内容

1. 页面结构和区块层级；
2. 页面之间的跳转关系；
3. 用户主任务路径；
4. 组件状态；
5. 表单规则；
6. 弹窗、抽屉、Toast、下拉菜单、标签页等交互；
7. 空状态、加载状态、错误状态；
8. 权限差异或角色差异；
9. 关键数据流；
10. 页面响应式布局。

### PRD 中的原型图表达

PRD HTML 中必须包含可视化的原型图 / 线框图 / 流程图表达，但要求如下：

- 优先复用 Demo 中的真实组件、状态机、流程步骤器或页面片段；
- 不允许单独画一套与 Demo 不一致的假线框；
- 不接受纯静态截图式图示；
- 如果使用流程图，优先使用 HTML + CSS + JS 实现可点击、可切换、可展开的交互流程模块；
- 如果使用 SVG 或 Mermaid，也必须补充点击、展开、切换或状态反馈；
- PRD 图示区与 Demo 区在流程顺序、按钮行为、状态枚举上必须一致。

## 八、阶段 4：生成 `02-产品需求文档.html`

创建独立的 PRD 可视化 HTML 页面。

### 视觉要求

PRD 页面必须采用金山风格的红白视觉体系：

- 主背景：白色或极浅灰；
- 主强调色：金山红 / 品牌红；
- 正文：深灰或近黑；
- 辅助色：浅红、暖灰、浅灰；
- 卡片：白底、细边框、轻阴影；
- 禁止大面积蓝紫渐变、黑金科技风、深色仪表盘风；
- 禁止把页面做成普通 Markdown 长文。

推荐色值：

```css
:root {
  --ks-red: #D71920;
  --ks-red-dark: #B5121B;
  --ks-red-light: #FFF1F2;
  --ks-text: #1F2329;
  --ks-muted: #646A73;
  --ks-border: #E5E6EB;
  --ks-bg: #F7F8FA;
  --ks-white: #FFFFFF;
}
```

### 页面结构建议

`02-产品需求文档.html` 至少包含：

1. 顶部 Hero：产品名称、定位、版本、核心价值；
2. 项目概览：背景、问题、解决方案、成功标准；
3. 用户与场景：目标用户、触发场景、用户故事；
4. 功能范围：P0 / P1 / P2 与明确不做；
5. 信息架构：模块关系、导航结构；
6. 页面清单：页面名称、目的、核心组件、入口出口；
7. 核心流程：用可交互流程图展示用户主路径；
8. 原型图 / 线框图：嵌入 Demo 中的关键页面片段或交互组件；
9. 交互规则：状态流转、异常处理、表单规则；
10. 数据与指标：核心数据对象、字段、验收指标；
11. 风险与待确认项：业务风险、技术风险、设计待确认点。

### 交互要求

PRD HTML 也应具备基础交互：

- 目录锚点跳转；
- 卡片展开 / 收起；
- 流程节点点击切换；
- 页面清单筛选或标签切换；
- 原型片段可点击或可切状态；
- 风险项可按类型筛选；
- 移动端可读。

## 九、阶段 5：生成 `03-高保真Demo.html`

创建独立的高保真 Demo HTML 页面。

### Demo 要求

Demo 必须是可演示原型，而不是静态页面。

必须满足：

1. 每个主要按钮可点击；
2. 每个导航入口有效；
3. 表单可以输入、校验和反馈；
4. 列表可筛选、切换或查看详情；
5. 弹窗、抽屉、Toast 等组件可触发；
6. 标签页、步骤条、状态切换可工作；
7. 空状态、错误状态、加载状态可被演示；
8. 页面之间无死链；
9. 文案全部为中文；
10. 适配桌面与移动端；
11. 可使用 `localStorage` 保存演示状态；
12. 所有 CSS 与 JS 默认写入同一个 HTML 文件中。

### Demo 视觉

Demo 的配色不强制必须红白。\
但如果用户没有指定产品风格，默认保持与金山红白体系协调，避免过度花哨。

如果产品题材适合图片，可以使用真实图片或高质量占位素材。\
如果无法联网或素材不可用，使用渐变、SVG、图标、插画式 Mock 图替代，但必须保持视觉完整。

## 十、阶段 6：生成 `04-PRD与Demo统一.html`

创建一个统一展示页，把 PRD 和 Demo 放在同一个 HTML 中。

### 统一页硬性要求

1. 顶部必须有两个主切换按钮：
   - `查看 PRD`
   - `查看 Demo`
2. 默认进入 PRD 视图；
3. 点击按钮后在同一页面内切换 PRD 和 Demo；
4. 不是简单上下拼接成长页面；
5. PRD 与 Demo 视图都能独立阅读和演示；
6. PRD 中的原型片段与 Demo 中对应页面逻辑一致；
7. 统一页中的样式、脚本、Mock 数据应尽量复用；
8. 不允许出现 PRD 说一种交互、Demo 做另一种交互的情况。

### 推荐结构

```html
<body>
  <header class="unified-shell">
    <button data-view="prd">查看 PRD</button>
    <button data-view="demo">查看 Demo</button>
  </header>

  <main>
    <section id="prdView">...</section>
    <section id="demoView" hidden>...</section>
  </main>

  <script>
    // 统一切换逻辑、共享 Mock 数据、共享状态工具函数
  </script>
</body>
```

## 十一、HTML 实现规范

所有 HTML 文件默认采用单文件实现：

- CSS 写在 `<style>` 中；
- JavaScript 写在 `<script>` 中；
- Mock 数据写在同一文件内；
- 不依赖构建工具；
- 不依赖后端；
- 如使用 CDN，只允许使用稳定、常见、可降级的外部资源；
- 关键功能不得依赖不稳定外部资源；
- 文件打开后即可演示。

### 基础工程要求

生成 HTML 时必须注意：

1. 语义化标签；
2. 清晰的 class 命名；
3. 组件化组织 CSS；
4. 事件绑定集中管理；
5. 避免重复脚本；
6. 避免无效按钮；
7. 避免控制台明显报错；
8. 移动端断点可用；
9. 中文字体栈合理；
10. 页面加载后默认状态完整。

## 十二、质量检查清单

宣布完成前，必须自检：

### 流程完整性

- 已完成需求头脑风暴；
- 已完成提示词增强；
- 已形成完整 PRD；
- 已设计原型图 / 线框图 / 交互规范；
- 已生成 3 个 HTML 交付物。

### PRD HTML

- 是否是红白金山风格；
- 是否不是普通 Markdown 长文；
- 是否包含完整 PRD 的关键结构；
- 是否包含信息架构、页面清单、流程图、原型图；
- PRD 中的图示是否具备交互；
- PRD 内容是否能支撑 Demo。

### Demo HTML

- 是否可独立打开；
- 是否具备高保真视觉；
- 是否所有主要按钮都有效；
- 是否有表单、弹窗、状态反馈；
- 是否有真实中文业务内容；
- 是否无死链、无空点击。

### 统一 HTML

- 是否有 `查看 PRD` 和 `查看 Demo` 两个按钮；
- 是否在同一页面内切换；
- 是否不是简单长页拼接；
- PRD 与 Demo 是否逻辑一致；
- 是否可直接用于汇报和演示。

## 十三、默认执行口径

当用户只提供一句产品想法时：

1. 先进行必要的需求澄清；
2. 本轮只输出澄清问题，不输出 PRD、不输出 Demo、不创建 HTML；
3. 等用户回答后，整理阶段 0 结论并向用户确认；
4. 用户确认阶段 0 结论后，才按完整流程执行；
5. 最终输出 3 个 HTML 文件；
6. 简要说明文件用途；
7. 不额外输出过程性 Markdown，除非用户明确要求。

当用户明确说“直接生成”“先按你的理解做”时：

1. 先检查阶段 0 通过标准是否已经满足；
2. 如果未满足，仍必须先问澄清问题，不得直接创建 HTML；
3. 如果已经满足，可以基于已确认信息执行；
4. 少量非关键假设必须体现在 PRD 的“待确认项”中；
5. 不得编造关键业务约束、用户角色、核心任务流或品牌事实。

当用户催促“别问了”“就按你想的做”但信息仍不足时：

1. 不能跳过阶段 0；
2. 可以提供 2-3 套产品方向让用户选择；
3. 用户至少选择或确认一个方向后，才能进入阶段 1。

当用户要求修改已有 HTML 时：

1. 优先读取已有文件；
2. 保持 3 个交付物之间的一致性；
3. 修改 Demo 交互后，同步修正 PRD 说明和统一页；
4. 修改 PRD 规则后，同步检查 Demo 是否需要调整。

## 十四、禁止事项

禁止以下行为：

- 只输出 PRD，不输出 Demo；
- 只输出统一页，不输出独立 PRD 和独立 Demo；
- 把头脑风暴和提示词增强完全省略；
- 把 PRD HTML 做成纯文字长文；
- 把 Demo 做成静态图片或静态页面；
- 在 PRD 里画一套与 Demo 不一致的假原型图；
- 使用蓝紫科技风覆盖 PRD 的红白金山风格；
- 使用 lorem ipsum 或明显占位文案；
- 留下“这里后续实现”“按钮暂不可用”等交付级缺口；
