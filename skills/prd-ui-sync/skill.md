# PRD-UI 增量同步 (prd-ui-sync)

PRD与UI原型的增量同步工具，支持标记变更模块，只重新生成变更部分，保留所有历史版本，永不覆盖原文件。

## 核心能力

- 识别PRD中的变更标记 `[CHANGED]` 或 `[NEW]`
- 模块级精准更新，不影响未变更部分
- 智能合并新旧UI，保持样式一致性
- **自动生成版本号文件，保留完整历史**
- 支持版本追踪和变更历史

## 使用场景

1. 用户修改PRD某模块后，只想更新对应UI部分
2. 新增功能模块，不改动现有UI
3. 修复PRD中的错误描述，同步到UI
4. 需要对比不同版本的PRD或UI
5. 需要回滚到之前的版本

## 文件命名规则（版本控制）

### 自动版本号生成

每次同步自动生成递增版本号：

```
workspace/
├── 02-产品需求文档.md              # 当前编辑中的PRD（输入）
├── 02-产品需求文档.v1.md           # 第1个版本（初始）
├── 02-产品需求文档.v2.md           # 第2个版本（首次修改）
├── 02-产品需求文档.v3.md           # 第3个版本（当前）
│
├── 03-线框图与交互规范.md           # 当前线框图
├── 03-线框图与交互规范.v1.md
├── 03-线框图与交互规范.v2.md
│
├── 04-UI交互原型.html              # 当前最新UI
├── 04-UI交互原型.v1.html           # 第1个版本UI
├── 04-UI交互原型.v2.html           # 第2个版本UI
├── 04-UI交互原型.v3.html           # 第3个版本UI（当前）
│
└── .prd-ui-sync/                   # 同步元数据目录
    ├── versions.json               # 版本记录
    └── changelog.md                # 变更日志
```

### 版本号文件格式

| 文件类型 | 命名格式 | 示例 |
|---------|---------|------|
| PRD | `02-产品需求文档.v{n}.md` | `02-产品需求文档.v3.md` |
| 线框图 | `03-线框图与交互规范.v{n}.md` | `03-线框图与交互规范.v2.md` |
| UI原型 | `04-UI交互原型.v{n}.html` | `04-UI交互原型.v5.html` |

### 特殊文件

- `xxx.md` / `xxx.html` - 永远是最新版本（软链接/副本）
- `xxx.latest.md` - 最新版本的显式标记
- `.prd-ui-sync/versions.json` - 版本元数据

## PRD 变更标记规范

### 标记语法

```markdown
## 模块名称 [CHANGED:2024-01-15]
模块内容...

## 模块名称 [NEW]
新增模块内容...

## 模块名称 [UNCHANGED]
保持不变的模块（可选标记，用于清晰）
```

### 模块结构示例

```markdown
# 产品需求文档

## 1. 执行摘要 [UNCHANGED]
...

## 2. 用户故事 [CHANGED:2024-01-15]
### 2.1 故事A
修改后的内容...

## 3. 功能规格
### 3.1 模块X [CHANGED]
- 功能点1（修改）
- 功能点2（新增）

### 3.2 模块Y [UNCHANGED]
...

### 3.3 模块Z [NEW]
全新模块内容...
```

## 工作流程

```
用户修改PRD → 标记变更模块 → 执行同步 → 智能合并 → 生成新版本文件
                                    ↓
                              保留旧版本文件
                                    ↓
                              更新版本元数据
```

## 执行步骤

### Step 1: 解析PRD，识别变更

```javascript
// 解析逻辑伪代码
const modules = parsePRD(prdContent);
const changedModules = modules.filter(m => 
  m.mark === '[CHANGED]' || m.mark === '[NEW]'
);
```

### Step 2: 确定新版本号

```javascript
// 读取版本记录
const versions = readVersions('.prd-ui-sync/versions.json');
const newVersion = versions.last + 1;  // 自动递增

// 记录变更
versions.history.push({
  version: newVersion,
  date: new Date().toISOString(),
  changedModules: changedModules.map(m => m.name),
  description: '用户描述的变更内容'
});
```

### Step 3: 分析原始UI结构

```javascript
// 基于上一版本分析
const previousUI = readFile(`04-UI交互原型.v${newVersion-1}.html`);
const uiStructure = parseUIStructure(previousUI);
```

### Step 4: 生成变更模块的UI

针对每个 [CHANGED] 或 [NEW] 模块：
1. 提取模块的PRD描述
2. 结合线框图（如有）
3. 生成独立的UI片段
4. 确保与原UI样式一致

### Step 5: 智能合并

```
上一版本UI ←——合并——→ 新生成UI片段
      ↓                    ↓
   保留未变更部分      替换/追加变更部分
      ↓                    ↓
         ↓ 输出生成新版本 ↓
   04-UI交互原型.v{n}.html
```

### Step 6: 输出新文件（不覆盖）

1. **保存新版本PRD**：`02-产品需求文档.v{n}.md`
2. **保存新版本UI**：`04-UI交互原型.v{n}.html`
3. **更新最新版软链接**：`04-UI交互原型.html` → `04-UI交互原型.v{n}.html`
4. **记录变更日志**：更新 `versions.json` 和 `changelog.md`
5. **可选**：更新PRD标记 `[CHANGED]` → `[SYNCED]`

## 用户操作流程

### 方式一：自动识别模式（推荐）

```bash
# 用户执行同步
/prd-ui-sync

# skill自动：
# 1. 读取当前PRD，查找所有变更标记
# 2. 确定新版本号（自动递增）
# 3. 只重新生成变更部分
# 4. 生成新版本的PRD和UI文件
# 5. 更新版本记录
```

**输出示例：**
```
✅ 检测到变更模块：Epic 2: 冥想室
📁 生成新版本：v3
📝 保存：02-产品需求文档.v3.md
🎨 保存：04-UI交互原型.v3.html
🔗 更新：04-UI交互原型.html → v3
📊 变更：冥想室模块（安慰功能 → 呼吸练习）
```

### 方式二：带描述的同步

```bash
/prd-ui-sync --message="将冥想室从安慰功能改为呼吸练习"

# 变更描述会记录到 changelog.md
```

### 方式三：指定版本号

```bash
/prd-ui-sync --version=5

# 强制使用指定版本号（如果冲突则提示）
```

### 方式四：预览模式

```bash
/prd-ui-sync --dry-run

# 显示将要生成的文件，不实际创建
# 输出：
# 将生成：02-产品需求文档.v3.md
# 将生成：04-UI交互原型.v3.html
# 变更模块：Epic 2: 冥想室
```

## 版本管理命令

| 命令 | 说明 |
|-----|------|
| `/prd-ui-sync` | 自动识别变更，生成新版本 |
| `/prd-ui-sync --list` | 列出所有版本历史 |
| `/prd-ui-sync --diff v2 v3` | 对比两个版本的差异 |
| `/prd-ui-sync --rollback v2` | 回滚到指定版本（生成回滚版本） |
| `/prd-ui-sync --mark-clear` | 同步后清除PRD变更标记 |

## 版本对比与回滚

### 查看版本历史

```bash
/prd-ui-sync --list

# 输出：
版本历史：
v3 (2024-01-15 14:30) - 将冥想室改为呼吸练习
v2 (2024-01-14 09:20) - 添加情绪日记图表功能
v1 (2024-01-13 16:00) - 初始版本
```

### 对比版本差异

```bash
/prd-ui-sync --diff v2 v3

# 输出差异报告：
## v2 → v3 差异
### PRD变更
- Epic 2: 冥想室 [CHANGED]
  - 移除：安慰功能、情绪接纳、拥抱按钮
  - 新增：呼吸练习、4-7-8呼吸、Box呼吸

### UI变更
- 修改：冥想室页面HTML结构
- 修改：冥想室JS交互逻辑
- 修改：CSS样式（琥珀色 → 青色）
```

### 回滚到历史版本

```bash
/prd-ui-sync --rollback v2

# 不删除v3，而是生成v4，内容与v2相同
# 输出：
# ✅ 已回滚到 v2 内容
# 📁 生成新版本：v4（基于v2）
# 📝 保存：02-产品需求文档.v4.md
# 🎨 保存：04-UI交互原型.v4.html
```

## 特殊情况处理

### 情况1：PRD无变更标记

```
检测到PRD无任何 [CHANGED] 或 [NEW] 标记
→ 提示用户：
  "未检测到变更标记，是否继续生成新版本？"
  选项：[是，基于当前内容生成v{n}] / [否，先标记变更]
```

### 情况2：版本号文件已存在

```
检测到 02-产品需求文档.v3.md 已存在
→ 自动使用下一个可用版本号：v4
```

### 情况3：磁盘空间不足

```
检查版本文件数量
→ 超过20个版本时提示：
  "已有{n}个历史版本，建议归档旧版本"
  选项：[继续] / [归档v1-v10] / [取消]
```

## 最佳实践

1. **小步修改** - 一次PRD变更控制在3-5个模块内
2. **及时同步** - 修改PRD后立即执行同步
3. **写变更描述** - 使用 `--message` 记录修改意图
4. **定期归档** - 版本过多时归档旧版本到子目录
5. **清理标记** - 同步后使用 `--mark-clear` 清理PRD标记

## 核心原则

1. **永不覆盖** - 所有历史版本永久保留
2. **自动版本** - 版本号自动递增，无需手动管理
