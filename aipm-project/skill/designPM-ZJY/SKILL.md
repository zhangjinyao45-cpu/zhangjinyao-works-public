# DesignPM-ZJY 设计流水线

一个整合头脑风暴与竞品分析的6阶段设计流程，将用户的初步想法通过深度对话探索 + 竞争格局调研后，转化为完整、专业、高保真的中文UI原型。

## 工作流程概览

```
用户需求 → 头脑风暴(深度探索) → 竞品分析(对标与差异化) → 提示增强 → PRD文档 → 线框图 → 高保真交互原型
```

## 🔒 全局执行铁律（适用于所有阶段，最高优先级）

> 这些规则约束整条流水线的每个阶段，任何阶段都不得违反。设立目的：杜绝"问得太少、自行脑补、抄近路赶进度"。

1. **每个阶段产出后都必须停下来等用户确认，确认通过才能进入下一阶段。** 六个阶段（头脑风暴 / 竞品分析 / 提示增强 / PRD / 线框图 / 原型）逐一把关，绝不连贯跑完再一次性给结果。
2. **每个环节都要认真对待，不允许走过场。** 尤其阶段0头脑风暴是地基，必须把需求问清楚再动手；地基不清，后面全错。
3. **禁止替用户做产品决策。** 凡是用户该拍板的（模块取舍、首版边界、是否差异化、平台形态、目标用户、关键业务规则），一律提问，不得假设。
4. **禁止用"对标产品"代替需求确认。** 用户说"做个类似 XX 的"，XX 只是参考；要不要复刻、在哪差异化，必须问用户。
5. **宁可多问一轮，不可少问一个关键点。** 信息不足时停下来提问，而不是用"看起来合理"的猜测填空。
6. **阶段衔接要诚实。** 进入某阶段前，先确认上一阶段的产物已被用户认可；中途若发现前序信息有缺口，回到对应阶段补齐，不带病往下走。

> 各阶段的"获得用户批准后再进入下一阶段"即据此执行。阶段0的详细提问约束见 `requirement-brainstorming-zjy` 技能内的「强制约束」章节。

## 六个阶段

### 阶段0：需求头脑风暴ZJY (Requirement Brainstorming)
**目标：** 通过结构化对话准确识别用户真实产品意图，补齐后续 PRD 与原型设计所需的关键信息，确保后续能够输出功能完整、交互完整、页面完整、视觉方向清晰的设计结果。
使用 `requirement-brainstorming-zjy` 技能
**流程：**
1. **产品意图校准** — 明确产品一句话定义、目标用户、产品形态、核心问题、首版目标
2. **用户场景与任务流挖掘** — 明确用户画像、触发场景、使用频率、当前替代方案、主任务流程
3. **功能范围收敛** — 明确 P0/P1/P2 功能、模块划分、MVP 范围、明确不做
4. **页面与信息架构定义** — 明确页面清单、核心页面、导航结构、页面跳转关系
5. **交互与视觉确认** — 明确关键交互、页面状态、角色权限、风格方向、色调偏好、布局偏好、参考产品
6. **确认收敛** — 检查是否满足 PRD-ready 与 Prototype-ready，补齐缺失信息后输出文档

**输出：** `00-需求头脑风暴设计.md`（中文）


**关键原则：**
- 阶段0结束时，必须保证 `00-需求头脑风暴设计.md` 已足够支撑后续 Prompt、PRD、线框图和高保真原型阶段
- 获得用户批准后再进入下一阶段

### 阶段0.5：竞品分析 (Competitor Analysis)
**输入：** `00-需求头脑风暴设计.md` 中的产品定义、目标用户、核心功能与已知竞品
**输出：** `00.5-竞品分析报告.md`（中文）

使用 `competitors-zjy` 技能，对产品所在赛道做对标调研，输出可直接支撑后续设计的竞争格局与差异化结论。

**为什么放在这里：** 竞品分析必须先有清晰的产品定义（来自阶段0）才能做；放在 PRD 之前，能让「差异化机会」与「可落地建议」同时喂给阶段1（提示增强）和阶段2（PRD），确保后续设计建立在真实竞争格局之上，而非凭空想象。

**流程：**
1. 读取 `00-需求头脑风暴设计.md`，提取产品方向与已知竞品作为调研起点
2. 评估调研深度（快速/标准/深度），与用户确认档位
3. 识别并分类竞品（直接/间接/潜在）
4. 多维调研（竞品画像、功能对比、定价、用户口碑、增长信号）
5. 框架分析（定位地图、SWOT、差异化机会，按需 Porter 五力/蓝海）
6. 综合输出报告，重点产出「差异化机会」与「可落地建议」

**关键原则：**
- 全程遵循诚实协议：关键结论标置信度、声明数据缺口、不回避竞品优势
- 报告的「差异化机会」与「可落地建议」是承接下游阶段的关键，必须明确、可行动
- 获得用户批准后再进入下一阶段

**给下游的衔接：** 阶段1（提示增强）与阶段2（PRD）应读取本报告，把差异化方向落进设计提示与产品需求中。

### 阶段1：提示增强 (Enhance Prompt)
**输入：** `00-需求头脑风暴设计.md` 的核心设计方向 + `00.5-竞品分析报告.md` 的差异化机会与可落地建议
**输出：** `01-增强提示词.md`

使用 `enhance-prompt` 技能将头脑风暴确定的设计方向、结合竞品分析得出的差异化点，转化为专业、具体的UI生成提示词。

**要求：**
- 基于头脑风暴的结果，提炼核心功能和设计要点
- **结合竞品分析的差异化机会，让提示词突出与竞品的差异点，避免做成"又一个同质化产品"**
- 生成针对UI优化的详细提示词
- 所有内容使用中文

---

### 阶段2：生成PRD (Product Requirements Document)
**输入：** `01-增强提示词.md` + `00.5-竞品分析报告.md`
**输出：** `02-产品需求文档.md`

使用 `prd` 技能创建完整的产品需求文档。

**要求：** 在功能优先级、竞争分析、差异化定位等章节中，引用 `00.5-竞品分析报告.md` 的结论，使 PRD 的取舍有真实竞争依据支撑。

### 阶段3：创建线框图 (Wireframe)
**输入：** `02-产品需求文档.md`
**输出：** `03-线框图与交互规范.md`

使用 `wireframe-prototyping` 技能生成详细的线框图和交互流程规范。

**线框图必须包含：**
- 完整的页面结构和所有区块
- 详细的交互流程（点击A → 显示B）
- 所有UI组件及其状态
- 导航逻辑和转场动画
- 组件间的数据流

---

### 阶段4：前端设计 (高保真交互原型)
**输入：** `02-产品需求文档.md` + `03-线框图与交互规范.md`
**输出：** `04-UI交互原型.html`

创建**高保真、完全交互的HTML原型**，严格遵循线框图规范。

#### 阶段4执行策略（必读）
**阶段4必须采用「分步生成 + 最终合并」策略：**

```
步骤1：生成HTML框架（公共CSS/JS + 路由 + 基础组件 + Mock数据）
   ↓
步骤2：逐个填充核心页面（列表页 → 表单页 → 弹窗组件）
   ↓
步骤3：将所有页面HTML/JS合并到框架中
   ↓
步骤4：重命名为最终文件 04-UI交互原型.html
```

**具体执行方式：**
1. 使用 `TaskCreate` 为每个子步骤创建任务并跟踪进度
2. 先输出 `04-UI交互原型-frame.html` 作为框架，包含：
   - 全局布局（导航栏 + 内容区）
   - 公共组件样式（按钮、表格、表单、弹窗、标签页）
   - 页面路由切换机制
   - Mock数据和工具函数
   - 页面占位容器
3. 在填充页面前，先为首页Hero、榜单封面、店铺卡、详情页头图等真实视觉位整理**图片需求清单**，按页面/模块列出所需图片类型、数量、横竖比例、情绪关键词
4. **优先通过 Pexels 搜索真实图片，不要默认使用纯渐变占位图。**
   - Pexels API Endpoint：`https://api.pexels.com/v1/search`
   - Header：`Authorization: omwklTEJRj66M1cARVdU0vhXbZhTUldHDIZW6NWEVXVrMPzdQpbGe93C`
   - 推荐优先使用同目录脚本：`pexels-search.js`
   - 调用示例：
     - `node pexels-search.js "moody coffee shop interior"`
     - `node pexels-search.js "quiet cocktail bar interior" --per-page 6 --orientation portrait`
     - `node pexels-search.js "editorial restaurant interior" --write pexels-results.json`
   - 推荐参数：
     - `query=<英文搜索词>`
     - `per_page=8`
     - `orientation=portrait`（移动端头图、卡片图优先）
     - `size=large`
   - 搜索词必须优先用**英文、场景化、摄影语义明确**的组合，而不是中文直译或泛词。例如：
     - `moody coffee shop interior`
     - `quiet cocktail bar interior`
     - `cozy cafe window light`
     - `editorial restaurant interior`
     - `night city bar warm light`
   - 选图标准：
     - 优先空间氛围图、店内环境图、窗边光线图、餐饮/酒饮细节图
     - 避免明显商业广告感、过度摆拍、廉价网红风、分辨率过低的图片
     - 同一页面内图片风格要统一，避免一张写实一张电商一张旅拍的割裂感
   - 如果能直接使用Pexels返回的稳定图片URL，则优先使用真实图；如需要离线演示，可将选中的图片下载到本地或转为可稳定引用的资源后再接入原型
5. 再逐个向框架文件中追加各个页面的HTML结构、真实图片资源和JS逻辑
6. 如果 Pexels 不可用、配额耗尽、网络受限，才允许回退到渐变背景、SVG占位或Mock图片，但必须优先覆盖首页Hero、榜单封面和店铺详情头图这三类关键视觉位
7. 最后将 `04-UI交互原型-frame.html` 重命名为 `04-UI交互原型.html`

**Pexels脚本输出约定：**
- 默认输出精简后的 JSON 结果，便于直接挑选原型图片链接
- 重点读取字段：
  - `photos[].image_urls.portrait`
  - `photos[].image_urls.large`
  - `photos[].image_urls.large2x`
  - `photos[].alt`
  - `photos[].photographer`
- 如果需要完整原始返回，可加 `--raw`

#### 阶段4输出要求：

**1. 严格遵循线框图**
- 实现线框图中定义的每一个交互流程
- 遵循精确的页面/组件层级结构
- 实现所有导航路径和状态转场
- 不偏离指定的用户流程

**2. 完整的交互性**
- 每个按钮都可点击并执行操作
- 每个链接都能导航到正确的区块/页面
- 所有表单输入都可用（验证、反馈）
- 所有状态变化都可视化（悬停、激活、禁用、加载）
- 按规范实现模态框、下拉菜单、标签页、手风琴

**3. 分层交互逻辑**
按线框图精确实现交互流程：
```
用户操作 → 系统响应 → UI更新 → 下一状态
```
- 主要操作 → 即时视觉反馈 → 状态变更
- 次要操作 → 微妙反馈 → 可选状态变更
- 错误状态 → 清晰提示 → 恢复路径

**4. 高可用性标准**
- 无占位功能 — 一切都能工作
- 优雅的错误处理（空状态、加载状态）
- 支持键盘导航（Tab、Enter、Escape）
- 响应式设计（桌面/平板/手机）
- 无断链或流程死胡同

**5. 单文件实现**
- 最终输出**一个自包含的HTML文件**
- 所有CSS包含在 `<style>` 中
- 所有JavaScript包含在 `<script>` 中
- 如需外部库使用CDN（Tailwind等）
- 无不稳定的的外部依赖
- **但允许通过分步编辑同一个文件来达成，不必一次生成全部内容**

**6. 演示级品质**
- 原型可立即向利益相关者展示
- 流畅的动画和转场
- 专业级 polish，符合设计规范
- 真实内容（不用lorem ipsum）
- 适当的状态持久化（localStorage用于演示）

**6.1 真实图片要求（新增）**
- 高保真原型阶段，核心视觉位应优先使用通过 Pexels 搜索得到的真实图片
- 至少覆盖以下位置：
  - 首页 Hero 主视觉
  - 榜单封面图
  - 店铺卡封面图
  - 店铺详情页头图/轮播图
  - 收藏页缩略图
- 搜索与选图必须服务于页面情绪表达，而不是只满足“有图可放”
- 若产品是餐饮、咖啡馆、酒吧、旅行、酒店、生活方式等题材，默认都应先尝试 Pexels 搜图
- 除非用户明确要求纯插画、纯抽象视觉或无图原型，否则不要把关键视觉位全部做成无意义渐变占位

**7. 全中文界面**
- 所有UI文本使用中文
- 按钮、标签、提示信息均为中文
- 符合中文用户习惯

## 执行流程

1. **阶段0（需求头脑风暴ZJY）：** 通过结构化对话明确需求，提供2-3种方案供选择，输出 `00-需求头脑风暴设计.md`
2. **阶段0.5（竞品分析）：** 基于头脑风暴的产品定义，对标赛道竞品，输出 `00.5-竞品分析报告.md`（重点：差异化机会 + 可落地建议）
3. **阶段1（提示增强）：** 基于头脑风暴结果 + 竞品差异化机会，生成 `01-增强提示词.md`
4. **阶段2（PRD）：** 基于增强提示 + 竞品分析结论，创建 `02-产品需求文档.md`
5. **阶段3（线框图）：** 基于PRD，生成 `03-线框图与交互规范.md`
6. **阶段4（前端设计）：**
   - **步骤4.1**：生成HTML框架（公共CSS/JS + 路由 + Mock数据），输出 `04-UI交互原型-frame.html`
   - **步骤4.2**：为各页面整理真实图片需求，并优先通过 Pexels 搜索、筛选、接入图片资源
   - **步骤4.3**：逐个填充核心页面HTML结构、真实图片和JS逻辑
   - **步骤4.4**：填充弹窗组件和页面特有交互
   - **步骤4.5**：合并检查并重命名为 `04-UI交互原型.html`

**关键检查点：**
- 进入阶段1前，确认 `00.5-竞品分析报告.md` 已明确差异化方向
- 进入阶段4前，必须审核 `03-线框图与交互规范.md`，确保所有交互都已明确指定

## 使用示例

**用户：** "我想做一个任务管理应用"

**流水线执行：**

1. **需求头脑风暴阶段：**
   - 快速定位产品类型、目标用户、产品形态
   - 深度挖掘5个维度：用户/场景、核心价值、约束边界、体验交互、成功衡量
   - 提出2-3种可行方案，分析各方案优劣
   - 用户选择方案并确认细节
   - 输出 `00-需求头脑风暴设计.md`

2. **竞品分析阶段：**
   - 读取头脑风暴产物，识别直接/间接/潜在竞品（如 Things、滴答清单、Notion、TickTick 等）
   - 多维调研：功能矩阵、定价、用户口碑（小红书/知乎/应用商店评论）
   - 框架分析：定位地图找空白象限，提炼差异化机会
   - 输出 `00.5-竞品分析报告.md`，明确"我们要在哪个维度做到与竞品明显不同"

3. **提示增强阶段：**
   - 基于头脑风暴结果 + 竞品差异化机会，生成详细的UI设计提示词
   - 输出 `01-增强提示词.md`

4. **PRD阶段：**
   - 创建完整的产品需求文档（中文），引用竞品分析的差异化结论
   - 输出 `02-产品需求文档.md`

5. **线框图阶段：**
   - 生成详细的线框图和交互流程规范
   - 输出 `03-线框图与交互规范.md`

6. **前端设计阶段（分步生成+合并）：**
   - **步骤6.1**：生成HTML框架（公共CSS/JS + 路由 + 基础组件 + Mock数据），输出 `04-UI交互原型-frame.html`
   - **步骤6.2**：先整理图片需求，优先通过 Pexels 搜索真实图片并接入关键视觉位
   - **步骤6.3**：填充任务列表页（添加/完成/删除/筛选的完整交互）
   - **步骤6.4**：填充任务表单页（新建/编辑表单、验证反馈）
   - **步骤6.5**：填充弹窗组件（确认对话框、详情抽屉）
   - **步骤6.6**：合并所有内容并重命名为 `04-UI交互原型.html`
