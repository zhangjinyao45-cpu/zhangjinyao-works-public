# 李航的工时估算方法论

## 核心原则

**工时用区间，不用单点。永远给"乐观/中位/悲观"三个数。**

## 估算公式

```
真实工时 = 开发 + 测试 + 数据迁移 + 文档 + 上线 + 缓冲
        ≈ 编码估算 × 1.6（缓冲系数）
```

## 不同复杂度的基准工时

| 类型 | 基准工时（中位） | 说明 |
|------|----------------|------|
| 静态页面（无交互） | 0.5-1 天 | 纯展示 |
| 简单 CRUD 页面 | 2-3 天 | 列表 + 增删改查 |
| 含复杂筛选的列表页 | 4-6 天 | 多 tag + 排序 + 分页 |
| 表单页（含验证） | 2-4 天 | 含字段联动 |
| 详情页 + 评论 | 3-5 天 | 含分页和懒加载 |
| 用户注册登录 | 3-5 天 | 含短信验证、第三方登录 |
| 支付流程 | 8-15 天 | 含资质 + 对接 + 测试 |
| 实时聊天 | 12-20 天 | WebSocket + 持久化 + 推送 |
| 后台数据看板 | 5-10 天 | 含图表和导出 |

> 以上基于 1 人前后端全栈，使用主流框架（React + Node/Python）

## 风险因子加权

如果遇到以下情况，工时翻倍：

| 风险因子 | 加权 |
|---------|------|
| 涉及第三方 API 对接 | × 1.5 |
| 涉及支付/资金/身份认证 | × 2.0 |
| 涉及合规（实名 / AIGC 标识） | × 1.5 |
| 涉及性能要求（大数据量 / 实时） | × 1.5 |
| 涉及多端同步（PC + 移动 + 小程序） | × 2.0 |
| 团队不熟悉的技术栈 | × 1.5 |
| 需求边界不清楚 | × 2.0（先把边界问清楚再估） |

## 输出格式

```markdown
## 工时估算

### MVP 总工时
- 乐观：30 天
- 中位：45 天
- 悲观：60 天
- 推荐周期：6-8 周（含 1 周缓冲 + 1 周联调测试）

### 单功能拆分
| 功能 | 乐观 | 中位 | 悲观 | 风险点 |
|------|------|------|------|--------|
| ... | ... | ... | ... | ... |

### 高风险点
1. ⚠️ [风险描述 + 建议替代方案]
2. ⚠️ ...

### 建议砍掉的功能
- [功能名]：[原因 + 砍掉的影响]
```

## 反对"PM 给死期"

```
❌ 错误："这个 5 天必须做完"
✅ 正确："这个最快 3 天，最慢 8 天，中位 5 天。
        如果一定要 3 天，要砍掉 [A] 和 [B] 两个细节。"
```

## 反对"自己拍工时"

```
❌ 错误："这个我估 10 天"（凭直觉）
✅ 正确："这个拆成 5 个子任务，分别是 1+2+1+3+2 天，
        再 × 1.6 缓冲系数 = 14 天，含联调测试。"
```
