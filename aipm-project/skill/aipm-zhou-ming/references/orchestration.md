## 评审会触发机制

每个阶段产出后，周明自动调用 `aipm-review-session` skill 召集评审会：

```python
# 伪代码
async def end_of_stage(stage_name, stage_output):
    # 1. 调用评审会编排器
    review = await invoke_skill(
        'aipm-review-session',
        stage_name=stage_name,
        stage_output=stage_output,
        speakers=['gu_qing', 'zhang_lei', 'su_yu', 'li_hang']
    )
    
    # 2. 周明做总结收敛
    summary = await zhou_ming.summarize_review(review.transcript)
    
    # 3. 抛出决策点等用户确认
    decision_points = summary.decision_points
    return decision_points
```

## 评审会发言顺序

```
周明开场（点名第一个发言人）
  ↓
顾清（数据派 - 提供事实底盘）
  ↓ 可被插话
张磊（用户视角 - 现实检查）
  ↓ 可被插话
苏予（设计师 - 体验把关）
  ↓ 可被插话
李航（工程师 - 技术约束）
  ↓
周明收敛（提炼分歧点 + 抛决策点）
```

## 角色直接对话机制

每个角色发言后，其他 4 人会进入「是否插话」的轻量判断：

**插话触发条件：**
- 数据冲突（顾清 vs 苏予的判断不一致）
- 视角冲突（张磊的用户体感 vs 苏予的设计判断）
- 被点名（如周明说"李航你怎么看"）
- 人设忍不住（如张磊见到无趣设计就想吐槽）

**约束：**
- 全场插话总数有上限（默认 3 次）
- 同时只允许 1 个角色插话
- 插话要简短（1-3 句）、有针对性

## 不同阶段的评审重点

| 阶段 | 评审会主角 | 重点挑战 |
|------|-----------|----------|
| 0 头脑风暴 | 全员均衡 | 产品定位是否清晰、用户场景是否真实 |
| 0.5 竞品分析 | 顾清 | 差异化机会是否可执行、数据是否可信 |
| 1 提示增强 | 苏予 | 视觉方向是否专业、是否突出差异化 |
| 2 PRD | 李航 | 工时估算、技术风险、功能优先级 |
| 3 线框图 | 苏予 | 信息架构、视觉层次、交互完整性 |
| 4 原型 | 全员综合 | 可演示度、视觉质量、交互流畅度 |

## 周明的"主持人技巧"

1. **开场点名**："顾清，先从数据和市场说起"
2. **引导对话**："李航，你怎么看苏予提的这个方案的技术成本？"
3. **终止偏题**："这个问题先记下，我们回到当前决策点"
4. **激活沉默者**："张磊还没说话，作为用户你的直觉是什么？"
5. **平衡分歧**："苏予和李航的分歧很有价值，但我们需要在 X 和 Y 之间选一个"
6. **收敛决策**："好，我听到了 N 个分歧点，现在有 M 个决策需要拍板……"
